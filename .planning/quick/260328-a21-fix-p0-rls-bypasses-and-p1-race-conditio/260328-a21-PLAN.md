<objective>
Fix P0 RLS bypasses and P1 race conditions identified in code review.

Three P0 RLS policy issues allow authenticated users to escalate privileges via PostgREST:
- trade_requests UPDATE policy lets participants mutate trade status directly
- trade_reviews INSERT policy lets users fabricate reviews for trades they never participated in
- user_rankings UPDATE policy lets users inflate their own scores

Two P1 race conditions:
- acceptTrade/declineTrade/cancelTrade/updateTradeStatus lack optimistic concurrency (completeTrade/skipReview already have it)
- consumeBackupCode has a TOCTOU race allowing double-use of a single code

Purpose: Close exploitable attack surface before launch.
Output: Hardened schema (requiring migration) and race-safe server actions.
</objective>

   ```sql
   reviewer_id = auth.uid()
   AND EXISTS (
     SELECT 1 FROM trade_requests tr
     WHERE tr.id = trade_id
       AND tr.status = 'completed'
       AND (tr.requester_id = auth.uid() OR tr.provider_id = auth.uid())
       AND reviewed_id = CASE
         WHEN tr.requester_id = auth.uid() THEN tr.provider_id
         ELSE tr.requester_id
       END
   )
   ```

<task type="auto">
  <name>Task 2: Add optimistic concurrency to trade actions and make consumeBackupCode atomic</name>
  <files>src/actions/trades.ts, src/lib/backup-codes.ts</files>
  <action>
**In `src/actions/trades.ts`:**

The `completeTrade` and `skipReview` functions already use optimistic concurrency (`.eq("status", trade.status).select("id").single()` on the update, with an error check on the result). Apply the same pattern to the four functions that lack it:

1. **`acceptTrade`** (line 141-148): Change the update call from:
   ```ts
   .update({ status: TRADE_STATUS.ACCEPTED, updated_at: ... })
   .eq("id", tradeId)
   ```
   to:
   ```ts
   .update({ status: TRADE_STATUS.ACCEPTED, updated_at: ... })
   .eq("id", tradeId)
   .eq("status", TRADE_STATUS.PENDING)
   .select("id")
   .single()
   ```
   Then check `if (updateError || !updated)` and return `{ error: "Trade was modified by another request" }`.

2. **`declineTrade`** (line 193-199): Same pattern. Add `.eq("status", TRADE_STATUS.PENDING)` and `.select("id").single()`. Check result. The status pre-check on line 188 (`if trade.status !== PENDING`) remains as an early fast-path but the WHERE clause is the actual concurrency guard.

3. **`cancelTrade`** (line 243-249): Same pattern. Add `.eq("status", TRADE_STATUS.PENDING)` and `.select("id").single()`. Check result.

4. **`updateTradeStatus`** (line 304-310): Same pattern. Add `.eq("status", trade.status)` (the current status from the prior read) and `.select("id").single()`. Check result. This function handles accepted->transferring and transferring->completed transitions, so the WHERE must use the dynamic `trade.status` value (not a hardcoded status).

For all four functions, destructure the update call as `const { data: updated, error: updateError }` matching the existing pattern in `completeTrade`.

**In `src/lib/backup-codes.ts`:**

5. **`consumeBackupCode`** (lines 87-115): The current flow is: read unused codes -> bcrypt compare each -> mark matched code as used. The race: two concurrent requests both read the same unused code, both verify it, both mark it used.

   Fix: Add `eq(backupCodes.used, false)` to the UPDATE WHERE clause so the update becomes:
   ```ts
   const result = await db
     .update(backupCodes)
     .set({ used: true, usedAt: new Date() })
     .where(and(eq(backupCodes.id, row.id), eq(backupCodes.used, false)))
     .returning({ id: backupCodes.id });
   ```
   Then check `if (result.length === 0)` -- this means another request already consumed this code between our read and update. In that case, `continue` to the next code in the loop (do not return success). If `result.length > 0`, the code was atomically consumed and we return success.

   This makes the operation atomic: the `WHERE used = false` in the UPDATE acts as a compare-and-swap. Only one concurrent request can succeed for the same code row.
