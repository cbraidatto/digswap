# Quick Task 260328-a21: Fix P0 RLS Bypasses and P1 Race Conditions Summary

**Closed 3 P0 RLS privilege escalation vectors and 2 P1 race conditions in trade actions and backup code consumption**

## Accomplishments
- Removed `trade_requests_update_participant` RLS policy that allowed authenticated users to mutate trade status directly via PostgREST
- Removed `user_rankings_update_own` RLS policy that allowed users to inflate their own ranking scores via PostgREST
- Hardened `trade_reviews_insert_own` RLS policy with correlated subquery enforcing: (a) reviewer is authenticated user, (b) trade is completed, (c) reviewer is a trade participant, (d) reviewed_id is the actual counterparty
- Added optimistic concurrency (compare-and-swap via WHERE status = expected) to acceptTrade, declineTrade, cancelTrade, and updateTradeStatus
- Made consumeBackupCode atomic against double-use via WHERE id = ? AND used = false with .returning() check

## Decisions Made
- Removed trade_requests UPDATE RLS policy entirely rather than restricting columns -- all trade_requests mutations go through admin client via Server Actions, no legitimate client-side UPDATE path exists
- Removed user_rankings UPDATE policy entirely -- rankings are managed server-side via raw SQL in completeTrade/skipReview
- Hardened trade_reviews INSERT with correlated subquery rather than a simpler check -- prevents fabricated reviews for trades the user never participated in
