**Fix 1 — Atomic counter increments (race condition):**

In both `completeTrade` and `skipReview`, replace the read-increment-write pattern for `trades_this_month` and `contribution_score` with atomic SQL increments. Instead of:
```typescript
const { data: sub } = await admin.from("subscriptions").select("trades_this_month").eq("user_id", userId).maybeSingle();
await admin.from("subscriptions").update({ trades_this_month: (sub?.trades_this_month ?? 0) + 1 }).eq("user_id", userId);
```
Use a single Supabase RPC call or, if RPC is unavailable, use the admin client's raw SQL capabilities. The simplest approach with the Supabase client is to use `.rpc()` for an increment function, but since we may not have an RPC function deployed, instead use a pattern that avoids the SELECT entirely:

For `subscriptions.trades_this_month`: Use admin client's `.rpc('increment_trades_this_month', { user_id_param: userId })`. Since we may not have this RPC, the pragmatic fix is to use a single UPDATE with a subquery-style approach. However, the Supabase JS client doesn't support `SET col = col + 1` directly.

**Pragmatic approach:** Replace the two separate loops (one for subscriptions, one for user_rankings) with single-query-per-user updates that use Supabase's `.rpc()` pattern. Create a small helper `atomicIncrement` that does the update in one call without a preceding read. Actually, the simplest Supabase-compatible fix: do the UPDATE directly without the SELECT by setting a default. Since `maybeSingle()` could return null (no row), the current code handles that with `?? 0`. The atomic equivalent: just do the UPDATE and let the database handle it. Use raw SQL via `admin.rpc` or accept that the Supabase client can't do `col = col + 1` and instead add an optimistic lock:

**FINAL APPROACH — Use Supabase `.rpc()` with a simple SQL function, OR more practically, batch both the select+update into a single query by removing the separate SELECT and relying on the fact that we can do `update().eq()` with a computed value. Since this requires a SQL function for true atomicity, create a single helper that wraps both operations with a transactional guarantee:**

The most reliable fix without deploying new SQL functions: keep the two-step pattern but add an optimistic concurrency version check. For each counter update:
1. SELECT the current value AND a version/updated_at
2. UPDATE with `.eq("updated_at", previousValue)` -- if another process updated between, the UPDATE affects 0 rows

However, the cleanest fix that works with the existing Supabase setup: use `admin.from("subscriptions").update({...}).eq("user_id", userId)` and compute the increment server-side by reading within the same server action invocation, but wrap in a serializable pattern. Since Next.js server actions are single-threaded per request but CAN run concurrently across requests:

**SIMPLEST CORRECT FIX:** Change the UPDATE queries to use raw SQL via `admin.rpc()` or Supabase's built-in capabilities. Since Drizzle is also available in this project, use `db.execute(sql\`UPDATE subscriptions SET trades_this_month = trades_this_month + 1 WHERE user_id = ${userId}\`)` via the Drizzle `sql` tagged template. This is truly atomic.

So: import `db` from `@/lib/db` and `sql` from `drizzle-orm`, then replace:
```typescript
// OLD (race-prone):
for (const userId of [trade.requester_id, trade.provider_id]) {
  const { data: sub } = await admin.from("subscriptions").select("trades_this_month")...;
  await admin.from("subscriptions").update({ trades_this_month: (sub?.trades_this_month ?? 0) + 1 })...;
}

// NEW (atomic):
for (const userId of [trade.requester_id, trade.provider_id]) {
  await db.execute(sql`UPDATE subscriptions SET trades_this_month = COALESCE(trades_this_month, 0) + 1, updated_at = NOW() WHERE user_id = ${userId}`);
}
```

Do the same for `contribution_score` in `user_rankings`:

**Fix 2 — IDOR / double-completion (optimistic concurrency):**

In `completeTrade`, the UPDATE at line 376 uses `.eq("id", tradeId)` but does NOT check the current status. Between the SELECT (line 345) and UPDATE (line 376), another request could have already set status to COMPLETED. Fix by adding `.eq("status", trade.status)` to the UPDATE and checking the result:

```typescript

  .from("trade_requests")
  .update({ status: TRADE_STATUS.COMPLETED, updated_at: new Date().toISOString() })
  .eq("id", tradeId)
  .eq("status", trade.status)  // Optimistic concurrency: only update if status hasn't changed
  .select("id")
  .single();

if (updateError || !updated) {

}
```
