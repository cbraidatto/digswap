## Decisions Made

- **Hypothesis C confirmed as root cause** (not A or B). Investigation evidence preserved in 01a/01b/01c. The plan author hedged across all three hypotheses; the migration intentionally covers all three (CREATE EXTENSION IF NOT EXISTS + defensive GRANTs + the wrapper) so it's safe to apply against prod regardless of which subset of issues exists there.
- **Migration SQL adjusted from plan recipe** because the actual `vault.create_secret` signature in supabase_vault v0.3.1 is `(text, text, text, uuid)` — 4 args including key_id — not the 3-arg signature the plan assumed. Plan's `GRANT EXECUTE ON FUNCTION vault.create_secret(text, text, text)` would have failed with `42883 function does not exist`. Used a DO block with pg_proc signature lookup to grant defensively only when the actual function exists with the expected signature.
- **Option B chosen for plaintext-row strategy** (vs Option A migrate-and-rewrap). Reasoning: plan author explicitly recommended Option B for lower blast radius (no plaintext OAuth token material handled in node process memory); Option A migrate-and-rewrap is also the path Phase 38 UAT will exercise end-to-end, so testing it via 2 manual back-fills here adds little signal. Both options achieve `plaintext_count = 0`. See "Deviations" below for why this was decided autonomously rather than waiting for user.
- **Vault errors surface as HTTP 500 JSON, not redirect to /error page.** Considered three options: (a) re-throw and let Next.js return its default 500 page; (b) redirect to a dedicated /error route; (c) return a NextResponse.json with explicit 500. Chose (c) because the Discogs callback is a programmatic OAuth callback consumed by the browser via redirect — a JSON 500 is the right wire-level signal for the user's redirected navigation to fail visibly, and it embeds a machine-readable error code (`discogs_vault_unavailable`) for support workflow.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Plan SQL had wrong `vault.create_secret` signature**
- **Found during:** Task 1 PHASE 1 investigation (Hypothesis B probe)
- **Issue:** Plan recipe said `GRANT EXECUTE ON FUNCTION vault.create_secret(text, text, text) TO service_role`. The actual signature in supabase_vault v0.3.1 is 4-arg `(text, text, text, uuid)`. Running the plan's SQL verbatim would fail with `42883 function vault.create_secret(text, text, text) does not exist`.
- **Fix:** Replaced direct `GRANT EXECUTE` with a DO block that looks up the actual function signature in `pg_proc` and grants only if a matching signature exists. Same pattern for `vault.update_secret` (real signature is 5-arg with secret_id uuid). Added `NOTIFY pgrst, 'reload schema'` so the new `public.vault_create_secret` wrapper is visible to PostgREST immediately rather than waiting up to 10 minutes for cache refresh.
- **Files modified:** supabase/migrations/20260424000000_enable_vault_extension.sql
- **Verification:** Migration applied cleanly to dev; smoke test (create + read + delete) PASS; idempotent re-run also clean.

**3. [Rule 2 - Missing critical] Outer catch in callback route was the FORBIDDEN log-and-redirect-to-success pattern**
- **Found during:** Task 3 STEP 4 (call-site audit, per Blocker 3 in plan context)
- **Issue:** `apps/web/src/app/api/discogs/callback/route.ts:169-174` (pre-fix) caught any thrown error, logged it, and returned `NextResponse.redirect('/settings?error=...')`. Per the plan's stated rule (and per appsec output contract), this is the FORBIDDEN log-and-redirect-to-2xx pattern: a Vault-storage failure (security-critical infrastructure incident) would have been UI-equivalent to "Discogs API momentarily down" (ordinary user retry case). After Task 3 STEP 2, `storeTokens()` now throws — without this fix, that throw would have been silently swallowed by the outer catch, completely undoing the hardening.
- **Fix:** Outer catch now inspects `error.message` for the `'Vault unavailable'` marker. If matched, returns `NextResponse.json({ ok: false, error: 'discogs_vault_unavailable', message: '...' }, { status: 500 })` — explicit non-2xx response. Other errors (Discogs API down, rate limit, network blips) preserve the prior /settings?error=... redirect since they are not security-critical.
- **Files modified:** apps/web/src/app/api/discogs/callback/route.ts
- **Verification:** Existing 3 callback integration tests still pass; call-site audit captured to evidence/01i with VERDICT line.
