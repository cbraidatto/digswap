## Objective

Build the web trade surfaces: `/trades` inbox and `/trades/[id]` detail page. Static server-rendered. Realtime and presence come in 15-03.

## Context

No `/trades` route exists today. The desktop handles all trade execution. The web surface is purely contextual: users see their trade history, read/send messages, and get a CTA to open the desktop app. Zero P2P in the browser.

Design language: retro/analog aesthetic consistent with the rest of the app (warm amber/brown palette, `#0d0d0d` backgrounds, serif-adjacent labels, mono data).

## Tasks

### Task 1 — /trades inbox page

`src/app/(protected)/trades/page.tsx`

- Server component
- Calls `listTradeThreads(userId)` from `src/lib/trades/messages.ts`
- Shows each trade as a row/card:
  - counterparty avatar + username
  - trade status badge
  - latest message preview (truncated, 80 chars)
  - unread count badge (amber, hidden if 0)
  - time of last activity
- Empty state: "No active trades — find records to trade on The Radar"
- Links to `/trades/[id]`

### Task 2 — /trades/[id] detail page

`src/app/(protected)/trades/[id]/page.tsx`

- Server component
- Fetches trade by id via Supabase server client
- Validates participation — redirect to `/trades` if user is not requester or provider
- Renders `TradeDetailHeader` + `TradeMessageThread` + `TradeMessageComposer`
- Calls `markTradeThreadRead` on load (server-side, silent)

### Task 3 — TradeDetailHeader

`src/app/(protected)/trades/[id]/_components/TradeDetailHeader.tsx`

Shows:
- Trade status (pill badge)
- Both legs: release title, artist, format, quality
- Counterparty card: avatar, username
- "Open in Desktop" CTA button — links to `digswap://trade/{tradeId}` (the protocol handler registered by the desktop app in Phase 17)
- Expiry timestamp if status is pending/lobby

### Task 4 — TradeMessageThread

`src/app/(protected)/trades/[id]/_components/TradeMessageThread.tsx`

- Client component (needs scroll-to-bottom behavior)
- Receives messages as prop from server component
- Groups consecutive messages from same sender
- System messages (kind: 'system'): centered, muted, italic — e.g. "Trade accepted", "Transfer complete"
- User messages: sender label + timestamp
- Own messages right-aligned, counterparty left-aligned
- Empty state: "No messages yet. Send the first one."

### Task 5 — TradeMessageComposer

`src/app/(protected)/trades/[id]/_components/TradeMessageComposer.tsx`

- Client component
- Textarea (max 2000 chars, char counter shown near limit)
- Submit via `sendTradeMessage` server action
- Disabled when trade status is terminal (completed/cancelled/expired/declined)
- Optimistic UI: append message immediately, revert on error
- `revalidatePath` in the action handles the server-side refresh

## Verification

- `/trades` loads without error for authenticated user
- `/trades/[id]` with valid participant: shows trade summary + CTA + thread
- `/trades/[id]` with non-participant: redirected or 404
- Composer submit inserts message and it appears in thread
- Terminal trade: composer is disabled
- "Open in Desktop" CTA points to `digswap://trade/{tradeId}`
