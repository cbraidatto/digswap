       - Props: `title` (string), `subtitle?` (string), `children` (ReactNode), `footer?` (ReactNode)

    3. **src/components/auth/social-login-buttons.tsx** -- Google and GitHub login buttons (per D-09, UI-SPEC Interaction States):
       - Two buttons: Google (with Google icon) and GitHub (with GitHub icon from Lucide)
       - Secondary button style: transparent bg, border, icon + label
       - Separated from main form by a `<Separator>` with "or continue with" text (per UI-SPEC Copywriting)
       - Each button calls a client-side function that invokes `supabase.auth.signInWithOAuth({ provider: "google" | "github", options: { redirectTo: "${SITE_URL}/api/auth/callback" } })`
       - Loading state: spinner replaces icon while OAuth redirect is pending
       - This is a client component ("use client")

    4. **src/actions/auth.ts** -- Server Actions for auth mutations (per 01-RESEARCH.md server action pattern, D-16, D-18):
       - `signUp(formData: FormData)`: Rate limit by IP -> validate with signUpSchema -> call `supabase.auth.signUp()` with emailRedirectTo -> return success/error. Per D-07, with "Confirm email" enabled, session is null after signup. Use generic error messages per OWASP (never reveal if email exists). Per 01-RESEARCH.md Server Action example.
       - `signIn(formData: FormData)`: Rate limit by IP -> validate with signInSchema -> call `supabase.auth.signInWithPassword()` -> check MFA level via `mfa.getAuthenticatorAssuranceLevel()` -> if AAL2 required, return `{ mfaRequired: true }` -> otherwise return success with redirect path. Record session in user_sessions table (for D-13 enforcement). If session count > 3, invalidate oldest via admin client.
       - `resendVerification(email: string)`: Rate limit by email -> call `supabase.auth.resend({ type: "signup", email })` -> return success message.
       - All actions use `"use server"` directive.
       - All actions get IP from `headers().get("x-forwarded-for")`.
  </action>
