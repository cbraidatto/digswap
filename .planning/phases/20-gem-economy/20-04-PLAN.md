    **File ownership and responsibility split:**
    - `apps/web/src/lib/gems/notifications.ts` -- Owns the **pure comparison logic** (`detectGemTierChanges`). Given two Maps of scores, returns which records changed tier. No database access, no side effects. This is the testable core.
    - `apps/web/src/app/api/discogs/import/route.ts` -- Owns the **orchestration**: (1) takes the pre-import snapshot query, (2) after import completes, takes the post-import snapshot query, (3) calls `detectGemTierChanges(preSnapshot, postSnapshot)`, (4) creates notification rows in the database for each change.
    - `apps/web/src/lib/discogs/import-worker.ts` -- NOT modified. The import worker processes individual pages and updates `rarity_score` on releases. The route.ts orchestrates the full job lifecycle and is the correct hook point for pre/post comparison.

    **1. Create `apps/web/src/lib/gems/notifications.ts` (pure function, no DB):**

    ```typescript
    export interface GemTierChange {
      releaseId: string;
      releaseTitle: string;
      discogsId: number | null;
      oldTier: GemTier;
      newTier: GemTier;
      oldWeight: number;
      newWeight: number;
    }

    export function detectGemTierChanges(
      oldScores: Map<string, { score: number; title: string; discogsId: number | null }>,
      newScores: Map<string, { score: number; title: string; discogsId: number | null }>,
    ): GemTierChange[]
    ```

    Logic: For each entry in newScores, if the entry exists in oldScores, compare `getGemTier(oldScore)` vs `getGemTier(newScore)`. If different, add to changes array. If entry does NOT exist in oldScores (new record), skip (no notification for initial add -- prevents first-sync flood per RESEARCH.md Pitfall 2).

    **2. Hook into import route (`apps/web/src/app/api/discogs/import/route.ts`):**

    After the final page is processed (when job is complete), add gem tier change detection:

    ```typescript

    // FROM collection_items ci JOIN releases r ON r.id = ci.release_id
    // WHERE ci.user_id = job.user_id
    // Build Map<releaseId, { score, title, discogsId }>
    ```

    b. **Post-import snapshot + diff** (after all pages processed and job completes):
    ```typescript
    // Same query as above (scores now updated by import-worker.ts)
    // Call detectGemTierChanges(preSnapshot, postSnapshot)
    // For each change, insert notification row:

    //   body: '{oldTierName} -> {newTierName} (+{weightDiff} gem value)' or '(-{weightDiff} gem value)'
    //   link: '/release/{discogsId}'
    ```

    c. Cap at 50 notifications per sync (same pattern as wantlist match notifications) to prevent notification flood on large collection changes.
