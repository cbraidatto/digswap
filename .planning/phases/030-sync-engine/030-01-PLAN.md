<objective>
Build the web-side sync infrastructure: schema migration for deletedAt, normalization functions, batch processor with dedup logic, and the API route that desktop calls.

Purpose: This is the server half of the sync engine. Desktop sends batches of track metadata here; the server deduplicates against existing Discogs releases, creates local-only releases for unmatched items, upserts collection items, and handles soft-deletion of removed files.

Output: Working POST /api/desktop/library/sync endpoint that processes track batches with full dedup and deletion support.
</objective>

    export function normalizeForDedup(value: string | null): string {
      if (!value) return "";
      return value
        .toLowerCase()
        .replace(/^(the|a|an)\s+/i, "")
        .replace(/[^\w\s]/g, "")
        .replace(/\s+/g, " ")

    `processSyncBatch(userId: string, tracks: TrackSyncPayload[], deletedPaths: string[]): Promise<SyncResponse>`

    Logic:
    1. **Group tracks by album key** using `makeAlbumKey(track.artist, track.album)`.
    2. **For each album group** (Stage 1 dedup per D-01):
       - Query `releases` table with Drizzle `db` client: `SELECT id, discogs_id FROM releases WHERE lower(artist) = ? AND lower(title) = ?` using normalized values.
       - If a release with `discogsId IS NOT NULL` is found: use that release ID (this is a Discogs match -> increment `linked` counter).
       - If a release with `discogsId IS NULL` is found: use that existing local release ID.
       - If no release found: INSERT a new release via `createAdminClient()` Supabase admin client with `{ title: album, artist: artist, year: year, discogs_id: null }`. Increment `created` counter.
       - Note: for dedup query, use the raw (un-normalized) artist+album for the release INSERT but the normalized form for the lookup. Actually, store the original casing in the release, but query using `lower()` + normalization for matching.
       - IMPORTANT: Do NOT use upsert on discogs_id for local releases (NULL != NULL pitfall). Use select-then-insert pattern.
    3. **For each track in the group**: Upsert a `collection_items` row via admin client:
       - Conflict resolution on `(user_id, release_id)` — if already exists, update `updatedAt`, `audioFormat`, `bitrate`, `sampleRate`, clear `deletedAt` to NULL (recovery case).
       - Fields: `userId`, `releaseId`, `addedVia: "local"`, `audioFormat: track.format`, `bitrate: track.bitrate`, `sampleRate: track.sampleRate`, `visibility: "not_trading"` (default for new items), `deletedAt: null`.
    4. **Low-confidence items** (per D-01): Tracks with `artistConfidence: "low"` OR `albumConfidence: "low"` that did NOT match an existing Discogs release get flagged. For now, just increment `dedupQueued` counter. The actual Discogs search queue is Phase 30 Plan 03 scope.
    5. **Handle deletedPaths** (per D-03): For each path in `deletedPaths`, query `collection_items` where `userId` matches and the associated release was created from a local track with that path. Since we don't store filePath on collection_items, we need a mapping strategy. **Approach:** Store `localTrackId` as `notes` field prefixed with `local:` (e.g., `notes = "local:{localTrackId}"`) to enable reverse lookup. Alternative simpler approach: the desktop sends the `releaseId` (web-side UUID) for deleted items if it tracked them. **Simplest approach per D-02:** Desktop sends `deletedPaths` which are file paths. Server cannot directly map file paths to collection items. **Resolution:** Add a `localPath` text column to `collection_items` (nullable, only populated for `addedVia = "local"` items). This enables deletion lookups. Wait — this adds schema complexity. **Better approach:** Desktop tracks `{ releaseId, userId }` in SQLite after successful sync. Desktop sends `deletedReleaseIds: string[]` instead of `deletedPaths`. The desktop side already knows which web release IDs correspond to deleted local files because it stores them after sync. Update the request interface:
       - Change `deletedPaths: string[]` to `deletedReleaseIds: string[]` (UUIDs of web-side release IDs to soft-delete).
       - Server: `UPDATE collection_items SET deleted_at = now() WHERE user_id = ? AND release_id IN (?) AND added_via = 'local'`.
    6. **Return** `SyncResponse` with all counters.

    ```typescript
    import { type NextRequest, NextResponse } from "next/server";
    import { z } from "zod";
    import { createAdminClient } from "@/lib/supabase/admin";
    import { apiRateLimit, safeLimit } from "@/lib/rate-limit";
    import { processSyncBatch } from "@/lib/sync/process-sync-batch";

    export const runtime = "nodejs";

    const trackPayloadSchema = z.object({
      localTrackId: z.string(),
      filePath: z.string(),
      artist: z.string().nullable(),
      album: z.string().nullable(),
      title: z.string().nullable(),
      year: z.number().nullable(),
      trackNumber: z.number().nullable(),
      format: z.string(),
      bitrate: z.number(),
      sampleRate: z.number(),
      duration: z.number(),
      artistConfidence: z.enum(["high", "low"]),
      albumConfidence: z.enum(["high", "low"]),
    });

    const syncBodySchema = z.object({
      tracks: z.array(trackPayloadSchema).max(100),
      deletedReleaseIds: z.array(z.string().uuid()).max(200),
    });

    export async function POST(request: NextRequest) {
      // 1. Bearer auth (same as handoff/consume)
      const authHeader = request.headers.get("authorization");
      const accessToken = authHeader?.startsWith("Bearer ")
        ? authHeader.slice("Bearer ".length)
        : null;
      if (!accessToken) {
        return NextResponse.json({ error: "Missing bearer token" }, { status: 401 });
      }

      const admin = createAdminClient();
      const { data: { user }, error: userError } = await admin.auth.getUser(accessToken);
      if (userError || !user) {
        return NextResponse.json({ error: "Invalid desktop session" }, { status: 401 });
      }

      // 2. Rate limit (apiRateLimit: 30/60s)
      const { success: rlSuccess } = await safeLimit(apiRateLimit, user.id, false);
      if (!rlSuccess) {
        return NextResponse.json({ error: "Too many requests" }, { status: 429 });
      }

      // 3. Parse + validate body
      let body: z.infer<typeof syncBodySchema>;
      try {
        const raw = await request.json();
        body = syncBodySchema.parse(raw);
      } catch (err) {
        const message = err instanceof z.ZodError
          ? err.issues[0]?.message ?? "Invalid request body"
          : "Invalid request body";
        return NextResponse.json({ error: message }, { status: 400 });
      }

      // 4. Process batch
      try {
        const result = await processSyncBatch(user.id, body.tracks, body.deletedReleaseIds);
        return NextResponse.json(result);
      } catch (err) {
        console.error("[sync] batch processing error:", err);
        return NextResponse.json(
          { error: "Internal sync error" },
          { status: 500 },
        );
      }
    }
    ```

    Key decisions:
    - `runtime = "nodejs"` because Drizzle postgres driver is not edge-compatible (same as OG image route, per Phase 10 decision)
    - `z.array(trackPayloadSchema).max(100)` enforces the 50-100 batch cap at 100 max (per D-02)
    - `deletedReleaseIds` max 200 to prevent abuse
    - Uses `apiRateLimit` (30/60s) — same tier as other API routes (per Phase 11 decision)
    - Error handling returns `ZodError.issues[0]` not `.errors[0]` (per Phase 13 decision)
