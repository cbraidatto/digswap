<objective>
Eliminate the nodeIntegration:true BrowserWindow security weakness by migrating the PeerJS bridge to an Electron utilityProcess.

Purpose: The current peer-session.ts creates a hidden BrowserWindow with nodeIntegration:true and contextIsolation:false to run PeerJS. While mitigated by loading only about:blank, this is architecturally weak — a utilityProcess is the proper Electron pattern for running Node.js code in a separate process without any renderer privileges. This closes audit finding #1.

Output: peer-session.ts refactored to use utilityProcess.fork(), new peer-bridge-worker.ts file as the forked script, electron.vite.config.ts updated with worker entry.
</objective>

**Step 1: Create apps/desktop/src/main/webrtc/peer-bridge-worker.ts**

This file is the script that runs inside the utilityProcess. It replaces the inline `buildPeerBridgeScript()` string that was previously injected via `executeJavaScript` into a BrowserWindow.

The worker script must:
- Import PeerJS via `require("peerjs")` (same pattern as the inline script: `PeerModule.default ?? PeerModule.Peer ?? PeerModule`)
- Receive initialization config via the first message from the parent: `{ type: "init", peerId, role, iceServers, expectedRemotePeerId }`
- Use `process.parentPort` for all IPC:
  - `process.parentPort.on("message", ({ data }) => { ... })` to receive commands
  - `process.parentPort.postMessage(event)` to send events back
- Implement the EXACT same command handlers as the current `buildPeerBridgeScript()`:
  - `connect` — sender initiates connection to expectedRemotePeerId
  - `send` — sends data via conn.send() with bufferedAmount backpressure check
  - `close-connection` — closes active connection
  - `destroy` — closes connection + destroys peer
  - `update-expected-peer` — updates expectedRemotePeerId variable
- Emit the EXACT same event types back to parent:
  - `peer-open` with `{ peerId }`
  - `connected` with `{ peerId: conn.peer }`
  - `disconnected`
  - `error` with `{ error: message }`
  - `data-json` with `{ payload }`
  - `data-buffer` with `{ payload: Buffer }`
  - `ice-candidate` with `{ payload: candidateType }`
  - `command-response` with `{ requestId, error?, payload? }`
- Include the same `wireConnection()` logic with expectedRemotePeerId identity check
- Include the same `startIceTracking()` logic for ICE candidate type detection
- Include the same `parseCandidateType()` helper
- Binary payloads: When receiving a "send" command with `kind: "buffer"`, reconstruct the Buffer from the serialized data. When emitting `data-buffer`, send the raw Buffer via postMessage (structured clone handles this).

The worker MUST NOT import `electron` or any renderer modules. It is pure Node.js + PeerJS.

**Step 3 (REQUIRED): Add peer-bridge-worker.ts as a Rollup input entry in electron.vite.config.ts**

`utilityProcess.fork()` requires a compiled .js file at the path specified. The electron-vite build for the `main` process currently has no explicit `rollupOptions.input` — it defaults to `src/main/index.ts`. The worker MUST be compiled as a separate entry point so it outputs as a standalone `.js` file.

In `apps/desktop/electron.vite.config.ts`, add `build.rollupOptions.input` to the `main` config:

```typescript
main: {
  plugins: [externalizeDepsPlugin({ exclude: ["@digswap/trade-domain"] })],
  build: {
    rollupOptions: {
      input: {
        index: resolveFromRoot("src/main/index.ts"),
        "peer-bridge-worker": resolveFromRoot("src/main/webrtc/peer-bridge-worker.ts"),
      },
    },
  },
  resolve: {
    alias: {
      "@main": resolveFromRoot("src/main"),
      "@shared": resolveFromRoot("src/shared"),
    },
  },
},
```

This ensures `out/main/peer-bridge-worker.js` is emitted alongside `out/main/index.js`. The `path.join(__dirname, "peer-bridge-worker.js")` reference in peer-session.ts will then resolve correctly because both files are in the same output directory.

After adding the entry, verify the build produces both files:
```bash
cd apps/desktop && npx electron-vite build 2>&1 | head -20
ls -la out/main/peer-bridge-worker.js
```

If the build output directory structure differs (e.g., nested), adjust the `path.join(__dirname, ...)` reference in peer-session.ts accordingly.
