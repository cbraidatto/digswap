# Plan 17-01: Monorepo Bootstrap + trade-domain Skeleton

## Goal
Convert the repository into a pnpm workspace monorepo, preserve the existing Next.js app under `apps/web`, scaffold `apps/desktop`, and create the initial `packages/trade-domain` package with types, enums, constants, and Zod schemas only.

## Context
Implements ADR-002 D-02 and D-03. This is the structural foundation for every subsequent desktop phase. The package boundary matters more than feature completeness here: `trade-domain` must stay pure and importable from both Electron and Next.js without pulling in framework or database code.

## Tasks

### Task 1: Bootstrap the workspace root

**Files:**
- `package.json`
- `pnpm-workspace.yaml`
- `tsconfig.base.json`
- `tsconfig.json`
- `biome.json`
- `.gitignore`

**Steps:**

1. Replace the root `package.json` with a workspace container package:
   - `private: true`
   - `packageManager: "pnpm@10.30.3"`
   - root scripts delegate to `@digswap/web`
   - add convenience scripts for `desktop`, `typecheck`, `lint`, and `test`

2. Create `pnpm-workspace.yaml` with:
   ```yaml
   packages:
     - apps/*
     - packages/*
   ```

3. Split TypeScript config:
   - `tsconfig.base.json` holds shared compiler options
   - root `tsconfig.json` becomes a lightweight references file for workspaces

4. Expand `biome.json` includes to cover `apps/**/*.ts(x)` and `packages/**/*.ts(x)`.

5. Update `.gitignore` for workspace-specific outputs:
   - `apps/*/.next`
   - `apps/*/dist`
   - `apps/*/playwright-report`
   - `apps/*/test-results`

### Task 2: Move the existing web app into apps/web

**Files/dirs moved or created:**
- `src` -> `apps/web/src`
- `public` -> `apps/web/public`
- `tests` -> `apps/web/tests`
- `scripts` -> `apps/web/scripts`
- `next.config.ts` -> `apps/web/next.config.ts`
- `next-env.d.ts` -> `apps/web/next-env.d.ts`
- `postcss.config.mjs` -> `apps/web/postcss.config.mjs`
- `components.json` -> `apps/web/components.json`
- `vitest.config.ts` -> `apps/web/vitest.config.ts`
- `playwright.config.ts` -> `apps/web/playwright.config.ts`
- `.env.local.example` -> `apps/web/.env.local.example`
- `package.json` content copied into `apps/web/package.json`
- `apps/web/tsconfig.json`

**Steps:**

1. Create `apps/web`.
2. Move the current app source, public assets, tests, and app-local tooling/config files into `apps/web`.
3. Create `apps/web/package.json` from the current app package definition:
   - rename to `@digswap/web`
   - keep all existing dependencies/devDependencies
   - add `typecheck` script
4. Create `apps/web/tsconfig.json` extending `../../tsconfig.base.json`:
   - preserve the `@/* -> ./src/*` alias
   - keep Next.js plugin
5. Keep backend infra that is not directly app runtime at root for now:
   - `drizzle/`
   - `supabase/`
   - `drizzle.config.ts`
6. Update moved configs if needed so they still target local `src`, `tests`, and `public`.

### Task 3: Scaffold apps/desktop placeholder

**Files:**
- `apps/desktop/package.json`
- `apps/desktop/tsconfig.json`
- `apps/desktop/src/index.ts`

**Steps:**

1. Create `apps/desktop` as a workspace package named `@digswap/desktop`.
2. Add placeholder scripts for `dev`, `build`, and `typecheck`.
3. Add `@digswap/trade-domain` as a `workspace:*` dependency.
4. Keep the placeholder implementation minimal:
   - no Electron runtime yet
   - just enough package structure for Phase 17-02

### Task 4: Create packages/trade-domain skeleton

**Files:**
- `packages/trade-domain/package.json`
- `packages/trade-domain/tsconfig.json`
- `packages/trade-domain/src/index.ts`
- `packages/trade-domain/src/constants.ts`
- `packages/trade-domain/src/status.ts`
- `packages/trade-domain/src/protocol.ts`
- `packages/trade-domain/src/runtime.ts`

**Steps:**

1. Create `@digswap/trade-domain` as an internal workspace package.
2. Keep it framework-free:
   - allowed imports: TypeScript, Zod
   - forbidden imports: Next.js, Drizzle, Supabase clients, React
3. Export the initial shared surface:
   - protocol version constants
   - trade status constants and transition map
   - transfer message types and Zod schemas
   - runtime/lease/receipt/handoff schemas and types
   - received-file-store convention constants only
4. Do not add implementation-heavy helpers in this phase.
