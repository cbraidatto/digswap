## 48h TTL mechanism (DEP-SB-07 lifecycle component)

Supabase Storage has NO native lifecycle/TTL feature. The 48h TTL is implemented as
a 3-piece mechanism (already in place after Plan 02 + Plan 03):

  1. trade_proposal_items.preview_expires_at column (set to now() + 48h on upload)
  2. pg_cron job 'trade-preview-cleanup' (hourly at :00) — invokes the Edge Function
  3. cleanup-trade-previews Edge Function — deletes objects whose preview_expires_at
     is in the past (.lte("preview_expires_at", nowIso))

This is verified end-to-end:
  - Plan 02 verified the 3 cron jobs are active under postgres role
  - Plan 03 verified Vault has both secrets so the cron's invoke function won't skip
  - Plan 04 verified the cleanup Edge Function returns 200 with the expected shape
    {"deleted":0,"bucket":"trade-previews","updated":0} on the empty database

The next 20:00 UTC tick will exercise the full pipeline (currently zero rows to
delete because the database is empty, but the mechanism is wired and live).
