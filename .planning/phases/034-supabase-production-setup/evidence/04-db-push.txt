## Halt-on-fail observations

- Migrations 33 + 35 failed on first attempt with `duplicate key on schema_migrations_pkey`.
  Root cause: MCP server auto-generates `version` from `now()` when filename has < 14-digit
  timestamp prefix. Multiple parallel apply_migration calls within the same second can
  collide on version. RESEARCH.md §10 catastrophic-halt protocol (drop+recreate) was NOT
  triggered because:
    a. PK violation = trail integrity intact, no partial schema state
    b. Sequential retry succeeded immediately on each
    c. Final list_migrations shows 35 unique entries, all 35 names recorded
- This is a known limitation of MCP-mode applying migrations whose names lack a
  full 14-digit timestamp prefix. Not a blocker for production correctness — the
  schema state matches dev exactly. Future migrations should use 14-digit prefixes
  to avoid the collision.
