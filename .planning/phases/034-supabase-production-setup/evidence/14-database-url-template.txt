## Required tokens (must all appear)

1. **Pooler host:** `aws-0-us-east-1.pooler.supabase.com:6543`
   - Region: us-east-1 (matches D-01 + matches Vercel iad1)
   - Port: 6543 (PgBouncer transaction-mode pooler — Pitfall #17)
   - NOT 5432 (direct connection — would exhaust serverless connection limits)

2. **Query string:** `?pgbouncer=true`
   - Tells the postgres driver this is a transaction-mode pooler
   - Disables prepared statement caching at the driver level (defensive — pairs with `prepare: false`)

3. **Drizzle config:** `prepare: false`
   - In `drizzle.config.ts` or wherever the postgres-js client is created:
     ```ts
     import postgres from 'postgres';
     import { drizzle } from 'drizzle-orm/postgres-js';

     const client = postgres(process.env.DATABASE_URL!, {
       prepare: false,  // <-- REQUIRED for PgBouncer transaction mode
     });
     export const db = drizzle(client);
     ```
   - Without `prepare: false`: queries silently fail with prepared statement errors after the first hot path

## Username structure (Supabase pooler quirk)

The pooler username has a TWO-PART structure: `postgres.<project_ref>`. So for digswap-prod:
- Username: `postgres.swyfhpgerzvvmoswkjyt`
- NOT just `postgres`

This is the pooler's tenant-routing scheme — the project_ref in the username tells PgBouncer which Supabase project to route to.
