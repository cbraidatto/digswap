# Workflow: Migration Review

**Objective:** Review a migration before it is applied to production. Produce a go/no-go recommendation with risk assessment.

## Trigger

The user provides a migration file (SQL or Drizzle schema diff) and asks whether it is safe to apply.

## Steps

### 1. Identify Changes
- Read the full migration SQL or Drizzle schema diff.
- Classify each statement: CREATE TABLE, ALTER TABLE, CREATE INDEX, CREATE/DROP POLICY, INSERT, UPDATE, DELETE, DROP, or custom function/trigger.

### 2. Check for Breaking Changes
Scan for red flags. Any match escalates the risk level:

| Red Flag | Risk | Why |
|----------|------|-----|
| `DROP TABLE` | CRITICAL | Irreversible data loss |
| `DROP COLUMN` | HIGH | Data loss, application code may reference the column |
| `ALTER COLUMN SET TYPE` | HIGH | Full table rewrite, exclusive lock on large tables |
| `ALTER COLUMN DROP NOT NULL` | MEDIUM | Application may assume non-null values |
| `ALTER COLUMN SET NOT NULL` | MEDIUM | Full table scan to validate, fails if NULLs exist |
| `RENAME TABLE` or `RENAME COLUMN` | HIGH | Breaks all references in application code and other policies |
| `DROP POLICY` without replacement | HIGH | Silently denies access if RLS is enabled with no remaining policies |
| `ALTER POLICY` changing USING clause | HIGH | May widen or narrow access unexpectedly |
| `TRUNCATE` | CRITICAL | Deletes all data without logging individual rows |

### 3. Assess Data Loss Risk
- Does the migration remove or overwrite any data?
- Is there a backup plan documented?
- Can the migration be reversed without data loss?

### 4. Analyze RLS Impact
- Does the migration add, remove, or modify any RLS policies?
- Does it create a new table? If yes, does the migration include `ENABLE ROW LEVEL SECURITY` and at least one policy?
- Cross-reference against the table categorization in `references/rls-audit.md`.

### 5. Verify Rollback SQL Exists
- The migration MUST include or be accompanied by rollback (DOWN) SQL.
- The rollback SQL must undo every change in the UP migration.
- If rollback SQL is missing, the review is an automatic NO-GO until it is provided.

### 6. Test on Branch Database
- Recommend testing on a Supabase branch DB or local Supabase instance before production.
- Check: does the migration apply cleanly? Does it break existing queries? Do RLS policies pass the 5-check test (anon, wrong user, owner, wrong INSERT, cross-boundary UPDATE)?

### 7. Check Drizzle Schema Sync
- If the migration adds columns, tables, or policies, verify that the corresponding Drizzle schema file in `apps/web/src/lib/db/schema/` has been updated.
- If the migration was generated by `drizzle-kit generate`, verify the generated SQL matches the intended schema change.

## Deliverable

Produce a migration review using `templates/migration-report.md` with:
- Risk level (LOW / MEDIUM / HIGH / CRITICAL)
- Summary of changes
- RLS impact assessment
- Rollback SQL verification
- Go / No-Go recommendation
- Conditions for Go (if conditional)
